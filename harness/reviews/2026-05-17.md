# chaextractor 하네스 검토

검토일: 2026-05-17

## 요약 판정

현재 하네스 수준은 "문서와 샘플은 있으나 실행 가능한 회귀 검증은 거의 없음"이다. `AGENTS.md`가 제품 명세 역할을 잘 하고 있고 Android 실제 ZIP 샘플이 `test/dataset/android/`에 남아 있다는 점은 좋은 출발점이다. 다만 명세, 구현, 이력 사이에 드리프트가 이미 생겼고, 이를 자동으로 잡아낼 장치가 없다.

가장 중요한 드리프트는 Windows 지원이다. 문서에는 macOS/Windows가 TODO로 남아 있지만, `index.html`에는 Windows 날짜/메시지/입장퇴장 패턴과 검증/파싱 코드가 이미 들어와 있다. 이 상태는 "지원하는 것처럼 보이나 보장되지 않는 기능"을 만든다.

하네스의 1차 목표는 기능을 더 구속하는 것이 아니라, 에이전트가 변경할 때 반드시 같은 기준선을 밟도록 만드는 것이다. 첫 검토에서는 실행 검증과 테스트 하네스에 무게를 두었지만, 프로젝트에 더 먼저 필요한 것은 "요구사항/표준/구현-only/미결정"을 나누는 계약 레이어다.

이 관점을 반영해 `/harness` 디렉터리를 추가했다.

- [../README.md](../README.md): 하네스의 역할과 운영 원칙
- [../MANIFEST.md](../MANIFEST.md): 표준, 요구사항, 구현-only, 미결정 항목의 현재 매니페스트

실행 테스트는 이 매니페스트가 잡은 경계 위에 쌓는 두 번째 단계가 되어야 한다.

## 보강 판정: 상위 표준 문서화 공백

기존 문서는 제품 기능과 도메인 패턴은 비교적 많이 담고 있으나, 보안/개인정보/아키텍처 표준을 별도의 계약으로 분리하지 않았다. 이 때문에 "구현되어 있다"와 "공식 요구사항이다"가 섞이고, 생성형 에이전트가 선의로 기능을 확장할 때 프로젝트 경계가 흐려질 수 있다.

특히 다음 표준은 `AGENTS.md`보다 상위 또는 별도 하네스 문서로 관리할 필요가 있다.

| 표준 영역 | 현재 문서화 수준 | 추가 판정 |
|---|---|---|
| 개인정보 경계 | "외부 전송 없음"으로 넓게 표현 | 폰트 CDN, 사용자 클릭 외부 링크, IndexedDB 캐시 경계를 분리해야 함 |
| 외부 네트워크 표면 | 문서화 없음 | 자동 요청과 사용자 클릭 링크를 매니페스트화해야 함 |
| XSS/입력 신뢰 경계 | 명시 없음 | 대화 내용/사용자명/파일명은 모두 untrusted로 표준화해야 함 |
| 캐시 정책 | IndexedDB 사용과 30일 정리 구현만 존재 | 민감 로컬 데이터, 삭제 UX, 캐시 키 정책을 요구사항으로 정의해야 함 |
| 구현-only 관리 | 없음 | Windows 후보 파서처럼 코드가 문서보다 앞선 항목을 분류해야 함 |
| 하네스 파일 추가 정책 | 없음 | 앱 런타임은 단일 HTML, 하네스/fixture/test는 별도 파일 허용으로 구분해야 함 |

## 기능별 검증 공백

### 파서

- iOS: 최소 fixture가 없다. 날짜 헤더, 24시간/오전오후 메시지, 사진, PDF, 누락 첨부파일, 연속 텍스트 병합을 고정 기대값으로 잡아야 한다.
- Android: 실제 ZIP 샘플은 좋지만 기대 JSON이 없다. 64자리 hash 사진, 연속 사진 줄, 입장 메시지 제외, 파일명 URL 인코딩을 검증해야 한다.
- Windows: 코드가 들어와 있으므로 방치하면 안 된다. 정식 지원 전이면 `windows-candidate` fixture로 "현재 실험 상태"만 고정하거나, 문서상 TODO와 맞추어 코드를 숨겨야 한다.
- URL 감지: `PATTERNS.URL`이 global regex(`/g`)라 `.test()` 호출 사이에 `lastIndex` 상태가 남을 수 있다. 링크 포함 여부는 회귀 테스트가 필요하다.
- 작은 파일: `validateChatFile`이 20줄 미만 파일을 무조건 거부한다. 실제 짧은 내보내기를 지원할지 결정하고 fixture로 고정해야 한다.

### 첨부파일

- iOS 매핑은 타임스탬프 기반으로 구현되어 있으나, 동일 시간대 다중 이미지/PDF의 기대 매핑이 고정되어 있지 않다.
- Android 이미지는 `attachment_ref` 직접 매핑이 가능하다.
- Android 일반 파일/PDF는 문서상 직접 매핑 가능하다고 되어 있으나 `isAttachmentFile()`이 hash 이미지와 iOS timestamp 파일만 인식한다. 샘플의 `파일: Signature%20iFD01_User%20manual.pdf` 같은 케이스는 실제 파일이 있어도 현재 매핑 대상에서 빠질 가능성이 높다.
- ZIP 모드와 폴더 모드의 attachment state가 새 업로드마다 초기화되는지 확인하는 테스트가 없다.

### UI/상태

- 검색 입력은 날짜 목록만 필터링한다. 메시지 리스트 하이라이트와 결과 스크롤은 구현되어 있지 않다.
- 날짜 목록/캘린더 동기화는 선택 흐름 중심으로 구현되어 있으나 캐시 hit, 검색 중 선택, 월 이동 경계 케이스가 검증되지 않는다.
- 리더 필터는 DOM display 토글 방식이다. 날짜 전환 후 필터 유지와 스크롤 마커의 의미가 검증되지 않는다.
- 모달은 클릭 닫기는 있으나 Escape 닫기와 포커스 복귀가 없다.
- 날짜 항목과 캘린더 day는 `div` 클릭 중심이라 키보드 접근성 요구와 거리가 있다.

### 캐시/성능/호환성

- IndexedDB cache hit의 날짜 정렬 회귀 가능성이 있다.
- 폴더 캐시 키는 첫 번째 대화 파일명/크기/mtime/파일 개수 중심이라 첨부파일 변경이나 추가 txt 변경을 충분히 반영하지 못할 수 있다.
- 50만 메시지 10초 요구는 문서에 있으나 재현 가능한 generator와 측정 스크립트가 없다.
- 브라우저 호환 요구는 있지만 Chrome/Safari/Firefox/Edge smoke matrix가 없다.

## 추천 하네스

### 1. Parser golden harness

목표: 플랫폼별 파싱 결과를 JSON 기대값으로 고정한다.

권장 구조:

```text
test/
  fixtures/
    android/
      TalkFile_KakaoTalk_Chats_2026-02-08_15.19.13_-131450398.zip
    ios-minimal/
      Talk_2026.1.27 21_37-1.txt
      20260127_213800_1.jpeg
      20260127_214000.pdf
    windows-candidate/
      KakaoTalk_20251109_2200_00_1_sample.txt
  expected/
    android-sample.json
    ios-minimal.json
    windows-candidate.json
  harness/
    parser-golden.md
```

검증 항목:

- 총 메시지 수, 날짜 수, 참여자 수
- 타입별 수: text/photo/file/emoticon
- 시스템 메시지 제외
- 같은 사용자 연속 텍스트 병합
- 첨부파일 매핑 수와 누락 파일 표시
- 리더 메시지 수
- 링크 감지

구현 방식 선택지는 두 가지다.

| 방식 | 장점 | 단점 | 추천 |
|---|---|---|---|
| 파서 코어를 별도 JS 파일로 추출 | 테스트가 가장 깨끗함 | 단일 HTML 원칙과 충돌 가능 | 장기 추천 |
| `index.html`에 test-only export 추가 | 단일 HTML 유지 | 전역 상태 관리 주의 필요 | 단기 추천 |

단기에는 `window.__CHAEXTRACTOR_TEST__ = { parseKakaoChat, parseMergedChatFiles, classifyContent, detectPlatform, mapAttachments }` 형태의 test-only hook을 두고 Playwright에서 브라우저 컨텍스트로 호출하는 방식이 가장 적다.

### 2. Browser smoke harness

목표: 실제 사용 흐름이 깨지는지 빠르게 확인한다.

Smoke 시나리오:

- 앱 첫 화면이 뜨고 ZIP/폴더 입력 UI가 존재한다.
- Android sample ZIP 업로드 후 시작 버튼이 활성화된다.
- 시작 후 날짜 목록과 채팅 헤더가 렌더링된다.
- 검색어 입력 시 날짜 목록이 줄어든다.
- 리더 필터 버튼 토글이 DOM 상태에 반영된다.
- 설정에서 테마/폰트 변경 시 `data-theme`, `data-font`, `localStorage`가 반영된다.
- 모바일 viewport에서 sidebar toggle/overlay가 동작한다.
- 이미지 클릭 시 image modal이 열리고 닫힌다.

### 3. Doc drift harness

목표: 문서와 구현의 기본 약속이 다시 벌어지는 것을 막는다.

체크 항목:

- `test/dataset` fixture가 README 또는 HARNESS 문서에 등재되어 있는지 확인한다.

### 4. Performance smoke harness

목표: 50만 메시지 요구를 바로 CI hard gate로 만들기보다, 성능 변화를 관찰 가능한 숫자로 만든다.

권장 단계:

- 1만/10만/50만 합성 iOS 텍스트 로그 generator를 만든다.
- 로컬 smoke는 1만 메시지 기준으로 빠르게 돌린다.
- 50만 메시지는 수동 또는 nightly 성격으로 측정하고 `HISTORY.md`에 결과를 누적한다.
- 측정값은 파싱 시간, messages 수, dates 수, peak attachment count 정도만 기록한다.
